# Feature Design：枪械弹道视觉重构

## 1. 元数据

- Owner：zoyluo
- Status：implemented，pending in-world visual QA
- Target version：2.2.0
- Related ADR：`ADR-0002-custom-ballistics-particles.md`
- Related issues：六类武器弹道粗糙、近距离发光十字遮挡准心、喷火器呈离散机关枪脉冲、枪械与弹药贴图细节不足

## 2. 目标与成功标准

- 六把枪拥有与用途和配色一致、容易辨认的枪口、弹道和命中特效。
- 第一人称枪口附近不生成大尺寸发光十字，准心和目标轮廓保持可读。
- 移除狙击枪命中后仍缓慢移动的假弹丸。
- 狙击枪、霰弹枪和 SMG 分别使用灰色实体、灰色范围和黑色实体语言；SMG 基础射速从每秒 5 发提升为每秒 10 发。
- 喷火器参考 Twilight Forest Hydra 的“枪口持续发射运动火焰粒子”原则形成连续火流。
- 轨道炮使用纯白细束，并在 `20` ticks（1 秒）内渐隐。
- 最大射程轨道炮的粒子生成调用不超过 `192`，相比旧实现降低至少 90%。
- 所有高频效果有常量预算和最大采样数。
- 空气和水中的弹道反馈不同。
- 六把枪和五种弹药的模型贴图严格采用确认参考图的轮廓、配色、像素描边与明暗层次。
- 构建、资源校验、Dedicated Server GameTest 和客户端资源加载通过。

## 3. 非目标

- 除 SMG 基础冷却由 `4` ticks 调整为 `2` ticks 外，不改变伤害、射程、散布、其他武器冷却、耐久、击退、爆炸强度和榴弹重力。
- 不修改现有 C2S Payload ID、字段或 `AdvancedWeaponFirePayload.Weapon` 顺序。
- 不加入后处理、屏幕震动、动态光源或配置界面。
- 不改变 registry item/entity ID、配方或存档数据。

## 4. 用户与玩法流程

- 单人玩家：开火后立即看到对应枪械的轻量枪口、弹道和命中反馈。
- 多人玩家：服务端广播相同的权威视觉终点，其他玩家也能看到弹道。
- 服务器管理员：无需安装额外依赖；粒子数量有硬上限。
- 其他 Mod：不新增 optional integration，不暴露内部视觉实现。

## 5. 视觉状态

| 武器 | 枪口 | 弹道 | 命中 |
| --- | --- | --- | --- |
| Sniper Rifle | 无发光十字 | 紧凑、不透明的灰色实体弹粒 | 少量灰色反弹粒子，方块命中补极少烟尘 |
| Shotgun | 无发光十字 | 7 条灰色扇形范围轨迹，每条最多 4 点 | 低透明灰色范围反馈，不形成烟墙 |
| Grenade Launcher | 橙色环形喷焰 | 可见榴弹、橙色余烬和稀疏烟带 | 中心闪光、冲击环、火花与爆炸烟 |
| SMG | 无发光十字 | 紧凑、不透明的黑色实体弹粒；基础冷却 2 ticks | 单个黑色实体反馈，避免每秒 10 发产生视觉噪声 |
| Flamethrower | 小尺寸火焰从枪口自然出现 | 每次 6 个带速度、散布和 14–18 tick 寿命的运动火焰，连续叠加成火柱 | 由火焰本体反馈，不额外铺静态线或烟环 |
| Railgun | 无收束大环 | 纯白细束，固定 20 tick 二次缓出渐隐 | 少量白色终点粒子，不生成青色冲击波 |

在水中，实体弹道和能量线使用气泡与弱化能量粒子；火焰粒子缩短寿命并以蒸汽/气泡代替远端火焰。

## 6. 模块影响

| 模块 | 变化 | 风险 |
| --- | --- | --- |
| registry | 在原七个 ID 基础上新增五个专用 Particle ID，共十二个 | 发布后必须稳定保留 |
| server visual | 新建有预算的弹道采样与效果编排 | 粒子调用数量 |
| client | 新增自定义 Sprite 粒子工厂 | 客户端类隔离 |
| resources | 新增粒子 JSON/纹理；重做枪械与弹药贴图 | 资源缺失或模型 UV 错误 |
| gameplay | 只替换视觉调用 | 不得改变伤害结果 |
| network | 复用原生粒子同步 | 无 Guns 协议变化 |

## 7. 数据与持久化

- 状态所有者：粒子为瞬态客户端状态；服务端不持久化。
- Schema 变化：无。
- registry ID：保留原七个 `guns:*` Particle ID；新增 `gray_round`、`gray_range`、`black_round`、`flame_jet`、`white_beam`。`blast_wave` 继续专用于榴弹，旧 `shockwave` ID 保留兼容但新轨道炮不再调用。
- 旧存档行为：完全兼容。

## 8. 服务端与客户端

- 服务端权威：命中终点、方块遮挡、实体伤害、爆炸和实际榴弹位置。
- 客户端：根据注册粒子类型执行缩放、淡出、旋转和纹理动画。
- 客户端预测：无；视觉由服务端事件产生。
- Dedicated Server：只加载 common 粒子注册和效果编排，不引用 `net.minecraft.client.*`。

## 9. 网络

- Payload：不新增 Guns Payload。
- 方向：Minecraft 原生服务器粒子同步。
- 限流：各武器效果具有硬采样上限；高频武器使用更低的单发预算。
- 协议兼容：现有客户端和服务端仍需安装同版本 Mod，但 Guns Payload 协议不变。

## 10. UI 与双语

本功能不新增玩家文本。`zh_cn.json` 与 `en_us.json` key 集保持一致，语言仍由 Minecraft 原生设置切换。

## 11. 性能

| 效果 | 单次/单 tick 上限 |
| --- | --- |
| Sniper | 80 个灰色实体采样点 |
| Shotgun | 7 条，每条最多 4 个灰色范围采样点 |
| SMG | 每发最多 12 个黑色实体采样点；基础每秒 10 发，最多 120 次粒子调用/秒 |
| Flamethrower | 每次 6 个运动粒子 |
| Grenade trail | 每 tick 3 个粒子 |
| Railgun | 最坏 172 次、硬上限 192 次生成调用；单粒子寿命 20 ticks |

- 不扫描额外实体或区块。
- 不强制加载区块。
- 不创建跨 tick 的静态轨迹队列。
- 采样算法为同步纯计算，复杂度由常量上限约束。

## 12. 兼容与 API

- 公共 API：不变。
- Mixin：不新增。
- 第三方依赖：不新增；Fabric API 已满足粒子注册与客户端工厂需求。
- 降级行为：资源异常时构建失败，不发布缺失粒子资源的 JAR。

## 13. 测试

- 单元级纯逻辑：线段采样数量、预算、零长度、最大长度、轨道炮最坏调用量和 20 tick 淡出契约。
- Gradle：粒子 ID、JSON、PNG、模型到贴图映射和参考图清单。
- GameTest：原有弹药、升级、配方与服务端行为继续通过。
- 手工：六把枪的第一/第三人称、空气/水下、最大射程和多人观察。

## 14. 发布与回滚

- Release 类型：Minor。
- Migration Notes：无存档迁移；客户端和服务端必须使用相同 JAR。
- 回滚：保留 Particle ID 注册并回退视觉编排；不回滚稳定内容 ID。

## 15. 未解决问题

- 开发客户端已完成资源加载且无 `guns:*` 模型、贴图或粒子错误；仍需在实际世界中手工确认六把枪的第一/第三人称、空气/水下、最大射程和多人旁观观感。
